**날짜**: 2026-04-22

**전제 버전**: 0.1.1 (pub.dev 배포 완료, 스크린샷 4장 등록)
**작업 위치**: `~/dev/byeonghopark/pixel_ui`

## 1. Scope & Goals

### 목표

pub.dev Platform 스코어 10 → 20 만점 달성을 위해 4개 플랫폼(Web / macOS / Linux / Windows) 공식 지원 추가.

### 성공 기준

- pub.dev `Platform` 칸에 Android / iOS / Web / macOS / Linux / Windows 6개 모두 표시
- 6개 플랫폼 모두 `flutter build` PASS (CI matrix로 증명)
- macOS · Web(Chrome)에서 본인 시각 실기 검증 PASS
- pub.dev Pub Points의 Platform 항목 10 → 20 만점 갱신

### 스코프 IN

- `example/`에 4개 플랫폼 scaffold 추가 (`web/`, `macos/`, `linux/`, `windows/`)
- example UX: `ConstrainedBox(maxWidth: 480)` + 중앙 정렬 (데스크톱·Web 가독성)
- `.github/workflows/build.yml` 신규 — matrix 빌드 전용
- README에 Platforms 섹션 + CI 빌드 배지 + "Web is smoke-tested on Chrome" 주의 명시
- `pubspec.yaml` version bump (0.1.1 → 0.2.0)
- `CHANGELOG.md` 0.2.0 엔트리 추가

### 스코프 OUT (다음 버전)

- analyze / test 일반 CI 워크플로우 → 0.3.0 (`ci.yml` 별도)
- 골든 테스트 (단일 macOS 호스트) → 0.3.0
- 브라우저 호환 매트릭스 (Safari, Firefox) → GitHub Pages Web 데모 도입 시점
- 반응형 레이아웃 재설계 (breakpoint 기반 grid) → 추후
- Mulmaru Mono 번들 → 0.4.0+
- example 데스크톱·Web 스크린샷 추가 → 시각 마케팅 단계

### 전제 (검증 완료)

`lib/` 코드는 다음 import만 사용:

- `package:flutter/widgets.dart` ✅ 전 플랫폼
- `package:flutter/rendering.dart` ✅ 전 플랫폼
- `package:flutter/foundation.dart` ✅ 전 플랫폼
- 패키지 내부 상호 import만 추가
- `dart:io`, `dart:html`, `MethodChannel`, 네이티브 의존성 **없음**

→ `lib/` 한 줄도 안 고쳐도 6개 플랫폼 빌드 통과 거의 확실. 0.2.0 작업 본질은 "lib 변경"이 아니라 **example scaffold + CI 검증**.

### 핵심 결정 근거 (브레인스토밍 합의 사항)

- **6개 플랫폼 전부**: Web만, Web+macOS 등 단계적 옵션이 있었으나 한 번에 +10 스코어 만점이 더 깔끔
- **CI 동시 도입**: cross-compile 불가 (Linux 빌드는 Linux 호스트, Windows 빌드는 Windows 호스트 필요) → CI 없이는 Linux/Windows 검증 자체가 불가능. "한 릴리스에 한 가지 변화" 원칙은 `build.yml`(이번)과 `ci.yml`(0.3.0) 분리로 유지
- **시각 검증은 macOS + Chrome만**: pixel_ui는 패키지지 앱이 아님. Safari/Firefox 호환성은 사용자(Flutter 개발자) 책임 영역. 5분 spot check가 release indecision 유발할 위험 회피
- **example UX는 ConstrainedBox 한 줄**: 반응형 재설계는 0.2.0 스코프 초과

### 2.3 example UX — ConstrainedBox + 중앙 정렬

`example/lib/main.dart`의 최상위 스크롤 위젯을 `Center` + `ConstrainedBox`로 한 단계 감싼다.

**구현 전 확인**: 실제 `main.dart` 구조를 먼저 읽어 어디를 감쌀지 정확한 위치 파악. 아래는 일반적인 패턴 예시 (실제 코드 구조에 맞춰 적용):

```dart
// 일반 패턴 — Before
Scaffold(
  body: SingleChildScrollView(
    child: Column(...),
  ),
)

// 일반 패턴 — After
Scaffold(
  body: Center(
    child: ConstrainedBox(
      constraints: const BoxConstraints(maxWidth: 480),
      child: SingleChildScrollView(
        child: Column(...),
      ),
    ),
  ),
)
```

**적용 원칙**: 가장 바깥 스크롤 컨테이너 한 곳만 감싼다. 내부 위젯 트리는 건드리지 않는다 — 픽셀 정렬·기존 레이아웃 회귀 위험 회피.

**효과**:

- 모바일 (iOS/Android): 화면 너비 < 480 → 변화 없음
- 데스크톱 (macOS/Linux/Windows) · Web: 콘텐츠가 480px 폭으로 중앙 정렬 → 모바일 쇼케이스 분위기 유지

### 2.4 영향 없는 영역

- 골든 테스트 (`test/goldens/`): 시뮬레이터 캡처가 아니라 위젯 직접 렌더 + 별도 ScreenshotFrame 사용 → ConstrainedBox 추가 영향 없음
- 이미 등록된 pub.dev 스크린샷 4장 (`example/assets/screenshots/*.png`): 골든 캡처와 무관 → 영향 없음

## 3. CI workflow — `.github/workflows/build.yml`

### 3.1 파일 신규 생성

```yaml
name: Platform Build Matrix

on:
  push:
    branches: [main]
  pull_request:

jobs:
  build:
    name: Build ${{ matrix.target }} on ${{ matrix.os }}
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false
      matrix:
        include:
          - os: ubuntu-latest
            target: web
            command: flutter build web
          - os: ubuntu-latest
            target: linux
            command: flutter build linux
            setup: sudo apt-get update && sudo apt-get install -y ninja-build libgtk-3-dev
          - os: ubuntu-latest
            target: apk
            command: flutter build apk --debug
          - os: macos-latest
            target: ios
            command: flutter build ios --no-codesign
          - os: macos-latest
            target: macos
            command: flutter build macos
          - os: windows-latest
            target: windows
            command: flutter build windows
    steps:
      - uses: actions/checkout@v4
      - uses: subosito/flutter-action@v2
        with:
          flutter-version: '3.32.7'
          channel: 'stable'
      - name: Platform setup
        if: matrix.setup
        run: ${{ matrix.setup }}
      - name: Pub get (root package)
        run: flutter pub get
      - name: Pub get (example)
        run: flutter pub get
        working-directory: example
      - name: Build
        run: ${{ matrix.command }}
        working-directory: example
```

### 3.2 설계 결정

- **`fail-fast: false`** — 한 플랫폼 실패해도 나머지 끝까지 진행. 어떤 플랫폼이 깨지는지 한 번에 파악
- **`flutter build apk --debug`** — release APK는 서명 키 필요. debug면 키 없이 통과
- **`flutter build ios --no-codesign`** — Apple Developer 계정 없이 통과
- **Linux setup** — `flutter build linux`는 GTK 3 · ninja 필요. ubuntu-latest 기본 미포함이라 설치 단계 명시
- **Web/macOS/Windows는 추가 setup 불필요** — flutter-action이 처리
- **루트 + example 양쪽 `pub get`** — example이 `path: ../`로 루트 pixel_ui 의존. 루트 `pub get` 누락 시 의존성 해결 실패

### 3.3 의도적 제외 (0.3.0 ci.yml로 이동)

- `flutter analyze`
- `flutter test`
- `dart pub publish --dry-run`
- 골든 테스트
- 캐싱 (`actions/cache`) — 빌드 시간 짧으니 일단 생략

### 3.4 검증

- 첫 PR(또는 main push)로 워크플로우 트리거
- 6개 job (web / linux / apk / ios / macos / windows) 모두 PASS 확인
- 실패 시: 로그 분석 → 수정 → 재 push 루프

## 6. 릴리스 시퀀스

### 6.1 작업 순서 (커밋 단위)

main 직접 작업 (개인 repo). 의미 있는 커밋 단위 분리:

```text
1. chore: scaffold web/macos/linux/windows in example
   → cd example && fvm flutter create --platforms=web,macos,linux,windows .
   → git status로 의도한 파일만 확인 후 add
   → .pubignore 재확인: example/web|macos|linux|windows가 아카이브에서
     제외되는지 의도대로인지 검증. 0.1.1에서 example/ 전체 제외 패턴이면
     자동 제외되므로 수정 불필요. 그렇지 않으면 .pubignore에 명시 추가.

2. feat(example): constrain showcase to 480px width on wide viewports
   → example/lib/main.dart의 최상위 스크롤 컨테이너에 Center + ConstrainedBox 래핑

3. ci: add platform build matrix workflow
   → .github/workflows/build.yml 신규
   → push 후 Actions 탭에서 6개 job PASS 확인

4. docs: add Platforms section and CI build badge to README
   → README.md 배지 + Platforms 섹션
```

### 6.2 배포 직전 (CI 6개 job PASS 확인 후)

```text
5. chore: bump version to 0.2.0
   → pubspec.yaml: version: 0.1.1 → 0.2.0

6. docs: changelog entry for 0.2.0 platform expansion
   → CHANGELOG.md 맨 위에:
     ## 0.2.0 — YYYY-MM-DD
     - Add official support for Web, macOS, Linux, Windows platforms.
     - Constrain example showcase to 480px max width on wide viewports.
```
